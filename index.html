HG Result System
Interactive PRD explainer for non-technical stakeholders

From spreadsheet-based results to a guided school result system.

This page explains what will be built, who will use it, how the screens connect, and the order the build should follow. It avoids developer jargon and turns the PRD into a visual walkthrough.

This single-page explainer translates the PRD into an implementation-facing stakeholder guide: architecture, role permissions, data model, validation gates, output requirements, and phased delivery plan.

12 secondary classes in scope: JSS 1–3 and SSS 1–3 across Art, Commerce, Science.
17 main app screens including login, score entry, admin screens, result generation, and profile password change.
10 permanent Google Sheet data tables, plus one Subjects Reference tab.

The simple story

Teachers enter small pieces of marks. The system checks completeness, calculates grades and positions, and only unlocks broadsheets and result slips when everything needed is ready.

Build roadmap

This is the practical order of work. Each phase produces something visible, testable, and understandable.

1

Set up the foundation

Create the Google Spreadsheet structure, the Apps Script project, and the first Super Admin account.

Create the 10-sheet schema, Subjects Reference tab, service-layer file structure, routing shell, and Super Admin bootstrap row with blank password hash and Force Password Change enabled.

What the school sees: the first login route exists and the Super Admin can set their own password on first use.
2

Build login and user management

Add username/password login, role checks, Add User, Edit User, Reset Password, and Change Password.

Implement password hashing, Force Password Change interception, employment status block, role-based routing, protected Super Admin row logic, and Admin/Super Admin user screens.

What to test: wrong username, wrong password, resigned account, first-time password change, Super Admin protection, and voluntary password change.
3

Connect classes, subjects, and students

Sync classes from the external school sheet, sync subjects from the internal reference tab, and cache student lists for fast mobile use.

Build Sync Classes, Sync Subjects, Class-Subject Assignment generation, student pull-on-open cache, manual Refresh Student List, and JSS/SSS filtering rules.

Why this matters: teachers should not manually rebuild class lists. The app must read the school’s real registration data.
4

Build teacher score entry

Teachers see their assignments, choose a score component, enter marks for the class, and save.

Implement Assignment List, Component Selector, Score Entry, component validation ranges, pre-fill existing entries, duplicate prevention, and Exam lock rules.

Key behaviour: marks are entered one component at a time, so phone entry is simpler and less error-prone.
5

Build Form Master tools

Form Masters manage class status, enter qualities, write remarks, and track class completion.

Implement Class Overview, Manage Class List, Student Term Status, PSQ Entry, Remarks Entry, lock reasons, and full-class access option.

Important decision: students marked Not Continuing remain visible for record history but are excluded from requirements and outputs.
6

Calculate results and unlock outputs

The system calculates totals, grades, comments, averages, class max, and positions, then unlocks broadsheets and slips only when complete.

Implement pure calculation functions for total, grade/comment, CLASS MAX, final average, ranking with tie handling, broadsheet readiness, and result-slip readiness.

Non-negotiable: buttons can be visible, but must stay disabled with clear reasons until requirements are complete.
7

Build exports and admin overview

Generate broadsheets, result slips, and an overview that shows which classes are ready.

Implement Broadsheet, Generate Results, PDF bundling, Excel export for broadsheet, Results Overview, Archive Session, and read-only historical access.

What the school gets: fewer manual checks, clearer readiness status, and safer result production.
8

Expose Power BI feed and launch pilot

Add read-only data endpoints for reporting, test with real users, then pilot with the secondary school.

Implement /api/scores, /api/students, /api/psq, /api/staff, denormalised records, user acceptance testing, performance checks, and launch support.

Future-proofing: Power BI uses the same contract that a later database-backed system can keep.

Rules, locks, and safety checks

These rules stop incomplete or incorrect results from being produced.

Score limits
Each component has a maximum mark. For example, Classwork is 0–6, Assignment is 0–2, Test 1 is 0–10, and Exam is 0–70.
Exam lock
Exam entry stays locked until every Active student has Test 1 and Test 2 for that subject. Exam Exempt and Not Continuing students are excluded from the lock count.
Broadsheet lock
Broadsheet export is disabled until all score components are complete for all Active students across all class subjects.
Result slip lock
Result generation is disabled until scores, PSQ ratings, and Form Master remarks are complete for every Active student.
Grade comments
Grades and comments are calculated automatically: A1 is Excellent, B2 is Very Good, through to F9 as Fail.
Position handling
Ties share the same position and the next rank is skipped. For SSS, ranking is across departments at the same level.

Outputs the school receives

The final outputs are protected by completion checks so the school does not accidentally publish incomplete results.

Student result slips

One PDF slip per student per term. Includes subject scores, total, grade, CLASS MAX, comment, final average, position, PSQ, remarks, next term fee, and resumption date.

PDF onlyBulk selected studentsLocked until complete

Broadsheet

A class-level summary table showing each student, subject totals, final average, grade, and position for the selected term and session.

Why this is useful later

If the system later moves from Google Sheets to a proper database, the Power BI dashboard can keep the same endpoint shapes. That means the report should not need a full rebuild.
